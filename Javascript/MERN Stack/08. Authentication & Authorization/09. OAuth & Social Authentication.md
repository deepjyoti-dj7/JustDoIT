# 09. OAuth & Social Authentication

## Introduction

OAuth 2.0 is an authorization framework that enables applications to obtain limited access to user accounts on third-party services (Google, Facebook, GitHub, etc.) without exposing passwords.

---

## Why OAuth?

### Problems with Traditional Authentication
❌ Users must create new accounts for every application
❌ Users must remember multiple passwords
❌ Applications store sensitive password data
❌ Users share passwords with third-party apps (security risk)

### Benefits of OAuth
✅ Users can sign in with existing accounts (Google, Facebook, etc.)
✅ No password sharing
✅ Delegated authorization
✅ Better user experience
✅ Enhanced security

---

## OAuth 2.0 Flow

### Authorization Code Flow (Most Common)

```
User                Client App           Authorization Server     Resource Server
 |                      |                        |                       |
 |-- Click Login ----> |                        |                       |
 |                      |-- Redirect to Auth --> |                       |
 |<-- Authorization Page ----------------       |                       |
 |                      |                        |                       |
 |-- Grant Permission > |                        |                       |
 |                      |<-- Authorization Code -|                       |
 |                      |                        |                       |
 |                      |-- Exchange Code -----> |                       |
 |                      |<-- Access Token -----  |                       |
 |                      |                        |                       |
 |                      |-- Request Data with Token ---------------->   |
 |<-- User Data ------- |<-- User Data -------------------------       |
```

## Google OAuth Implementation

### 1. Setup Google OAuth Credentials

1. Go to [Google Cloud Console](https://console.cloud.google.com/)
2. Create a new project
3. Enable Google+ API
4. Create OAuth 2.0 credentials
5. Add authorized redirect URI: `http://localhost:5000/api/auth/google/callback`

### 2. Install Dependencies

```bash
npm install passport passport-google-oauth20 express-session
```

### 3. Environment Variables

```env
GOOGLE_CLIENT_ID=your_client_id
GOOGLE_CLIENT_SECRET=your_client_secret
GOOGLE_CALLBACK_URL=http://localhost:5000/api/auth/google/callback
```

### 4. Passport Configuration

```javascript
// config/passport.js
const passport = require('passport');
const GoogleStrategy = require('passport-google-oauth20').Strategy;
const User = require('../models/User');

passport.use(
  new GoogleStrategy(
    {
      clientID: process.env.GOOGLE_CLIENT_ID,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET,
      callbackURL: process.env.GOOGLE_CALLBACK_URL
    },
    async (accessToken, refreshToken, profile, done) => {
      try {
        // Check if user already exists
        let user = await User.findOne({ googleId: profile.id });
        
        if (user) {
          // User exists, return user
          return done(null, user);
        }
        
        // Create new user
        user = await User.create({
          googleId: profile.id,
          name: profile.displayName,
          email: profile.emails[0].value,
          avatar: profile.photos[0].value,
          provider: 'google'
        });
        
        done(null, user);
      } catch (error) {
        done(error, null);
      }
    }
  )
);

// Serialize user for session
passport.serializeUser((user, done) => {
  done(null, user.id);
});

// Deserialize user from session
passport.deserializeUser(async (id, done) => {
  try {
    const user = await User.findById(id);
    done(null, user);
  } catch (error) {
    done(error, null);
  }
});

module.exports = passport;
```

### 6. Routes

```javascript
// routes/auth.js
const express = require('express');
const router = express.Router();
const passport = require('../config/passport');
const jwt = require('jsonwebtoken');

// @route   GET /api/auth/google
// @desc    Initiate Google OAuth
router.get('/google',
  passport.authenticate('google', {
    scope: ['profile', 'email']
  })
);

// @route   GET /api/auth/google/callback
// @desc    Google OAuth callback
router.get('/google/callback',
  passport.authenticate('google', { 
    failureRedirect: '/login',
    session: false 
  }),
  (req, res) => {
    // Generate JWT token
    const token = jwt.sign(
      { id: req.user._id },
      process.env.JWT_SECRET,
      { expiresIn: '7d' }
    );
    
    // Redirect to frontend with token
    res.redirect(`${process.env.CLIENT_URL}/auth/success?token=${token}`);
  }
);

module.exports = router;
```

### Auth Success Handler

```javascript
import { useEffect } from 'react';
import { useNavigate, useSearchParams } from 'react-router-dom';

const AuthSuccess = () => {
  const [searchParams] = useSearchParams();
  const navigate = useNavigate();
  
  useEffect(() => {
    const token = searchParams.get('token');
    
    if (token) {
      // Store token
      localStorage.setItem('token', token);
      
      // Redirect to dashboard
      navigate('/dashboard');
    } else {
      // No token, redirect to login
      navigate('/login');
    }
  }, [searchParams, navigate]);
  
  return <div>Logging you in...</div>;
};

export default AuthSuccess;
```

## Linking Accounts

Allow users to link multiple OAuth providers to one account.

```javascript
// controllers/authController.js
exports.linkGoogleAccount = async (req, res) => {
  try {
    // User is already logged in (req.user from JWT)
    const { googleId, email } = req.body;
    
    // Check if Google account is already linked to another user
    const existingUser = await User.findOne({ googleId });
    if (existingUser && existingUser.id !== req.user.id) {
      return res.status(400).json({
        success: false,
        message: 'This Google account is already linked to another user'
      });
    }
    
    // Link Google account
    await User.findByIdAndUpdate(req.user.id, {
      googleId,
      email // Update email if different
    });
    
    res.json({
      success: true,
      message: 'Google account linked successfully'
    });
  } catch (error) {
    res.status(500).json({
      success: false,
      message: error.message
    });
  }
};
```

## Handling Email Conflicts

```javascript
// In OAuth strategy
async (accessToken, refreshToken, profile, done) => {
  try {
    const email = profile.emails[0].value;
    
    // Check if user exists with this Google ID
    let user = await User.findOne({ googleId: profile.id });
    
    if (user) {
      return done(null, user);
    }
    
    // Check if user exists with this email
    user = await User.findOne({ email });
    
    if (user) {
      // Email exists but not linked to Google
      // Option 1: Auto-link (risky)
      user.googleId = profile.id;
      await user.save();
      return done(null, user);
      
      // Option 2: Return error (safer)
      // return done(new Error('Email already registered. Please login with password.'), null);
    }
    
    // Create new user
    user = await User.create({
      googleId: profile.id,
      name: profile.displayName,
      email,
      avatar: profile.photos[0].value,
      provider: 'google',
      emailVerified: true // Trust OAuth provider
    });
    
    done(null, user);
  } catch (error) {
    done(error, null);
  }
}
```

## Interview Questions

**Q1: What is OAuth 2.0?**
- Authorization framework for delegated access
- Allows third-party apps to access user data without password
- Uses access tokens instead of credentials

**Q2: What's the difference between OAuth and OpenID Connect?**
- **OAuth 2.0**: Authorization (what you can access)
- **OpenID Connect**: Authentication layer on top of OAuth (who you are)

**Q3: What is an access token?**
- Credential used to access protected resources
- Short-lived, revocable
- Issued after successful authorization

**Q4: What are OAuth scopes?**
- Permissions requested by application
- Examples: `profile`, `email`, `read:users`
- User must grant these permissions

**Q5: How do you handle OAuth email conflicts?**
- Check if email already exists
- Auto-link if email matches (risky)
- Or require manual account linking (safer)

**Q6: What's the difference between authorization code and implicit flow?**
- **Authorization Code**: Most secure, uses backend, exchanges code for token
- **Implicit**: Less secure, client-side only, token in URL (deprecated)

## Best Practices

1. **Use Authorization Code Flow** - Most secure for web apps
2. **Validate redirect URIs** - Prevent open redirect vulnerabilities
3. **Handle email conflicts** - Check for existing accounts
4. **Store provider info** - Track which provider user used
5. **Allow account linking** - Let users link multiple providers
6. **Trust OAuth providers** - Mark email as verified
7. **Implement logout** - Revoke tokens properly
8. **Use HTTPS** - Always in production
9. **Handle errors gracefully** - Provider downtime, denied permissions
10. **Keep secrets secure** - Environment variables, never commit

---

## Summary

- **OAuth 2.0** enables third-party authentication
- **Authorization Code Flow** most common for web apps
- Use **Passport.js** for easy OAuth integration
- Support **multiple providers** (Google, Facebook, GitHub)
- Handle **email conflicts** when users have existing accounts
- Generate **JWT token** after OAuth success
- **Redirect to frontend** with token in URL
- Store **provider ID** to identify OAuth users
- Allow **account linking** for multiple providers
- **Trust OAuth providers** for email verification
