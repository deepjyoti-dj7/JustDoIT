## What are Starters?

**Spring Boot Starters** are dependency descriptors that bundle a collection of related dependencies together. They provide a one-stop solution for including necessary dependencies for specific functionality.

### Core Concept
> Instead of adding 10+ dependencies for a web application, just add `spring-boot-starter-web` and get everything you need!

## Why Starters?

### Without Starters (Traditional Spring)
```xml
<dependencies>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>5.3.9</version>
    </dependency>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-web</artifactId>
        <version>5.3.9</version>
    </dependency>
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-webmvc</artifactId>
        <version>5.3.9</version>
    </dependency>
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
        <version>2.12.3</version>
    </dependency>
    <dependency>
        <groupId>org.apache.tomcat.embed</groupId>
        <artifactId>tomcat-embed-core</artifactId>
        <version>9.0.50</version>
    </dependency>
    <!-- 15+ more dependencies... -->
    <!-- Version management nightmare! -->
</dependencies>
```

### With Starters (Spring Boot)
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!-- Single dependency, no version needed! -->
</dependency>
```

## Benefits of Starters

1. **Dependency Management** - No need to manage versions manually
2. **Compatibility** - Pre-tested dependency combinations
3. **Simplicity** - One dependency instead of many
4. **Convention** - Sensible defaults included
5. **Auto-configuration** - Works with Spring Boot auto-configuration
6. **Consistency** - Same versions across all Spring dependencies

## Naming Convention

**Pattern:** `spring-boot-starter-{name}`

- **Official Starters:** `spring-boot-starter-*`
- **Third-party Starters:** `{name}-spring-boot-starter`

**Examples:**
- `spring-boot-starter-web` (Official)
- `mybatis-spring-boot-starter` (Third-party)

## Creating Custom Starter

### Step 1: Create Maven Project
```xml
<artifactId>my-custom-spring-boot-starter</artifactId>
```

### Step 2: Add Dependencies
```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-autoconfigure</artifactId>
    </dependency>
</dependencies>
```

### Step 3: Create Auto-configuration
```java
@Configuration
@ConditionalOnClass(MyService.class)
@EnableConfigurationProperties(MyProperties.class)
public class MyAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean
    public MyService myService(MyProperties properties) {
        return new MyService(properties);
    }
}
```

### Step 4: Create Properties
```java
@ConfigurationProperties(prefix = "myservice")
public class MyProperties {
    private String apiKey;
    private int timeout = 5000;
    // Getters and setters
}
```

### Step 5: Register in spring.factories
```properties
# src/main/resources/META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.example.MyAutoConfiguration
```

### Step 6: Usage
```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>my-custom-spring-boot-starter</artifactId>
    <version>1.0.0</version>
</dependency>
```

```properties
myservice.api-key=abc123
myservice.timeout=10000
```

## Version Management

### Parent POM Approach
```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
</parent>

<!-- No version needed for starters -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
```

### BOM Approach (Without Parent)
```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
```

## Best Practices

1. ‚úÖ **Use official starters** - Well-tested and maintained
2. ‚úÖ **Include only needed starters** - Avoid unused dependencies
3. ‚úÖ **Use parent POM** - For consistent version management
4. ‚úÖ **Check transitive dependencies** - Use `mvn dependency:tree`
5. ‚úÖ **Exclude specific dependencies** - When needed
6. ‚úÖ **Use scope appropriately** - test, runtime, provided
7. ‚ùå **Don't specify versions** - Let parent POM manage
8. ‚ùå **Don't add individual Spring libs** - Use starters instead

## Interview Questions

**Q: What are Spring Boot Starters?**  
A: Dependency descriptors that bundle related dependencies together with compatible versions.

**Q: Why use starters?**  
A: Simplify dependency management, ensure compatibility, reduce configuration.

**Q: Difference between spring-boot-starter and spring-boot-starter-web?**  
A: `spring-boot-starter` is core (logging, auto-config), `spring-boot-starter-web` adds web capabilities (MVC, Tomcat).

**Q: Can we create custom starters?**  
A: Yes, by creating auto-configuration and registering in spring.factories.

**Q: How to change embedded server?**  
A: Exclude default (Tomcat) and add desired server (Jetty/Undertow) starter.

**Q: Do starters contain implementation code?**  
A: No, they're just dependency descriptors that pull in actual libraries.

## Key Takeaways

- Starters = Dependency bundles
- One starter replaces 10+ dependencies
- Versions managed by parent POM
- Auto-configuration works with starters
- Custom starters possible
- Follow naming convention: `spring-boot-starter-{name}`
- Use official starters when available
